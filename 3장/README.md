<aside>
💡 입력한 코드를 “읽고(Read), 해석(Eval)하고, 반환(Print)하고, 종료할 때까지 반복(Loop)한다” 의 줄임말

</aside>

맥에서 REPL 사용할 수 있는 방법

- 맥 자체 터미널
- vscode의 터미널

_REPL 종료 방법 : `.exit`_

## Module이란?

<aside>
💡 특정한 기능을 하는 함수나 변수들의 집합 → 브라우저의 자바스크립트와 차이점이다!

</aside>

- 모듈은 자체로도 하나의 프로그램이면서, 다른 프로그램의 부품으로도 사용 가능하다.
- 하나 만들어두면, 여러 프로그램에서 재사용이 가능하다.
- 파일 하나가 모듈 하나가 되며, 파일별로 코드를 모듈화할 수 있다.
- 노드에서 사용하는 모듈 형식
  - **CommonJS** Module
    - 표준 자바스크립트 모듈은 아니지만 노드 생태계에서 가장 널리 쓰이는 모듈
  - **ECMAScript** Module
    - 공식적인 자바스크립트 모듈 형식
    - 확장자명이 js 대신 `mjs`(js 확장자에서 ES모듈을 사용하려면 package.json에 `type: “module”` 속성을 부여)
    - import는 `Promise`를 반환 → await나 then을 붙임
- 모듈을 불러올때는 require 함수를 사용한다.
- **순환 참조(Circular dependency)** : 모듈이 서로를 require하고 있는 구조 → 에러 발생하지 않고 빈 객체로 변경됨
- **다이나믹 임포트(Dynamic import)** : 조건부로 모듈을 불러오는 것

> 궁금한 점 : CommonJS 모듈과 ECMAScript 모듈 중에 어떤 방식을 많이 사용할까?

## Node 내장 객체

<aside>
💡 노드에서는 브라우저의 window와 같이 따로 설치하지 않고 바로 사용 가능한 내장 객체와 내장 모듈이 존재한다.

</aside>

노드 프로그래밍을 할 때 많이 쓰이는 내장 객체들에 대해서 알아보자

- global
  - globalThis : 브라우저 환경과 노드 환경을 모두 아우르는 객체(브라우저 : window, 노드 : global)
- console
  - 보통 디버깅 용도로 많이 사용
  - console. + time, log, error, table, dir, trace
  - 더욱 더 다양한 메소드들이 추가되고 있는 중이며, 실제로는 이것들보다 훨씬 다양한 메소드들이 존재한다.
- Timer

  - setTimeout(), setInterval(), setImmediate()
  - 타이머는 콜백 기반의 API이지만, 프로미스 방식을 사용할 수도 있다.

- Process
  - 현재 실행되고 있는 노드 프로세스에 대한 정보를 담고 있다.
  - 이 기록을 작성하고 있는 중의 나의 노드 프로세스 정보
  - process.env : 노드의 환경 변수
  - process.nextTick : 다른 콜백함수들보다 먼저 실행됨
  - process.exit : 실행 중인 노드 프로세스를 종료

## OS

<aside>
💡 OS 모듈은 주로 컴퓨터 내부 자원에 빈번하게 접근하는 경우에 사용된다.

</aside>

웹 브라우저에서 사용되는 자바스크립트는 운영체제에 대한 정보를 가져올 수 없지만,

노드의 OS 모듈은 운영체제에 대한 정보를 가져올 수 있다.

내장 모듈인 os를 불러오는 방법 : `require(’os’)` or `require(’node:os’)`

일반적인 웹 서비스를 만들 때는 잘 사용되지는 않지만,

운영체제별로 다른 서비스를 제공하고 싶을 때 유용하게 사용될 것이다.

## Path

<aside>
💡 폴더와 파일의 경로를 쉽게 조작하도록 도와주는 모듈이며, 운영체제별로 경로 구분자가 다르다.

</aside>

path모듈의 경로구분자는 크게 윈도우 타입과 POSIX 타입으로 구분한다.

- Window type : `\`
- POSIX type : `/`

노드는 require.main 파일을 기준으로 상대경로를 인식한다.

따라서 require.main과는 다른 디렉터리의 파일이 상대경로를 가진다면 의도와는 다르게 동작할 가능성이 있다.

- **상대경로** : 현재 파일이 기준이 된다. → 같은 경로 : `.` & 다른 경로 : `..`
- **절대경로** : 루트 폴더나 노드 프로세서가 실행되는 위치가 기준

<br>

## URL

<aside>
💡 인터넷 주소를 쉽게 조작하도록 도와주는 모듈이며, 요즘은 WHATWG 방식을 많이 사용한다.

</aside>

_WHATWG : 웹 표준을 정하는 단체의 이름_

URL의 구조

- href
  - origin
    - protocol
    - username
    - password
    - host
      - hostname
      - port
  - pathname
  - search(쿼리 스트링) : 주소를 통해 데이터를 전달할 때 사용(**searchParams**라는 특수한 객체 생성)
  - hash

이렇게 분해된 URL을 다시 원래 상태로 조립하기 위해서는? → `url.format()` 사용

`searchParams` 객체에 대해서 조금 더 정리를 해보자

| getAll(key) | 키에 해당하는 모든 값을 가져옴 | append(key, value) | 해당 키와 값을 추가                    |
| ----------- | ------------------------------ | ------------------ | -------------------------------------- |
| get(key)    | 키에 해당하는 첫번째 값 가져옴 | set(key, value)    | 같은 키의 값들을 모두 지우고 새로 추가 |
| has(key)    | 키가 있는지 없는지 검사        | delete(key)        | 해당 키를 제거                         |

## dns

<aside>
💡 DNS(Domain Name System)을 다룰 때 사용하는 모듈이다.

</aside>

주로 도메인을 통해서 IP나 기타 DNS 정보를 얻고자 할 때 사용한다.

보통 사용되는 레코드들

- A, AAAA : ipv 주소
- NS : 네임 서버
- SOA : 도메인 정보
- CNAME : 별칭
- MX : 메일 서버

이 파트에 대해서 좀 더 찾아보다가 **DNS 증폭 공격, Ddos 공격**등에 대해서 찾아보다가 머리가 터지는 줄 알았읍니다.

## crypto

<aside>
💡 다양한 방식의 암호화를 도와주는 모듈이다.

</aside>

고객의 비밀번호같은 중요한 정보는 반드시 암호화해야 한다.

안전 장치를 이중으로 만들어두는 것이 좋다.

<br>

### 단방향 암호화

Q. What? A. 복호화(암호화된 문자열을 다시 원래대로 돌려놓는 것)할 수 없는 암호화 방식

단방향 암호화 알고리즘은 주로 해시 기법을 사용한다.

<br>

### 양방향 암호화

Q. What? A. 암호화된 문자열을 복화할 수 있으며, 키(key)라는 개념이 사용된다.

암호화에 대해서 코드를 완벽하게 이해하려면 암호학을 추가적으로 공부하는 것이 좋다.

지금은 그냥 개념에 대해서만 아는 것으로도 충분!

## Util

<aside>

- `util.promisify` : 콜백 함수를 프로미스 패턴으로 바꿔준다.(async, await 패턴 사용이 가능하다.)

## 파일 시스템 접근하기

fs 모듈은 파일 시스템에 접근하는 모듈이다. → 파일을 생성, 삭제, 읽기, 쓰기가 가능하다.

- fs.readFile()의 결과물은 `버퍼(buffer)`라는 형식으로 제공
- 버퍼는 메모리의 데이터라고 생각하면 되고, 사람이 읽을 수 있는 형식이 아니다 → `toString()`으로 변환
- fs는 기본적으로 콜백 형식이기 때문에 `프로미스 형식`으로 바꿔주는 방법을 사용
- `readFile()` : 파일을 읽기, `writeFile()` : 파일을 생성하고 작성하기

## 비동기 메소드 & 동기 메소드

<aside>
💡 노드는 대부분의 메소드를 비동기 방식으로 처리하지만 몇몇 메소드는 동기 방식으로 사용 가능하다.

</aside>

동기와 비동기 : 백그라운드 작업 완료 확인 여부

- 동기 : 해당 파일의 작업이 모두 끝나야 다음 작업으로 넘어감(작업 완료 확인 O)
- 비동기 : 해당 파일의 작업을 요청해두고 다음 작업으로 넘어감(작업 완료 확인 X)

노드의 동기 메소드들은 이름 뒤에 sync가 붙어 있어 구분하기 쉽다(ex. `readFile` vs `readFileSync`)
